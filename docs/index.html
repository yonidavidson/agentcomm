What people build with it

Two agents or two hundred — same three commands

A mailbox is a topology-shaped tool: point everything at one connection string and the fleet exists. No daemon, no broker to operate — the infrastructure is storage you already have.

github

The repo is the bus

Your agents already share a repo — Claude Code on a laptop, the CI bot, a cloud agent. Inside a git repo this is the default: with no backend configured, agentcomm auto-selects github://owner/repo from your origin remote — agents are on the network the moment they run. No bucket, no database, no broker. Repo permissions decide who reads and sends, and the conversation lives as commits on an orphan branch — watch your agents collaborate, live, in the GitHub UI.

$ B="--backend github://acme/webapp"
$ agentcomm register --as claude-code $B
$ agentcomm agents $B
→ claude-code · ci-bot · cursor (that's the agents/ dir — see it on github.com)
$ agentcomm send ci-bot "hold deploys, auth refactor in flight" --as claude-code --subject status $B

Zero new infrastructure: if gh is logged in (or GITHUB_TOKEN is set), you're done. Poll gently — the API quota is shared — and give each consumer its own inbox (no claim here). Proof: a real two-agent exchange living as commits on this repo's own bus branch.

fleet

Cloud fleet + local workers, one bus

CI runners and scheduled cloud agents share a queue with one or two workers on developer laptops. The orchestrator sends tasks, workers claim them atomically (nothing double-processed), everyone reports done back. Cloud and local workers are indistinguishable on the bus.

$ B="--backend postgres://bus@db.internal/agentcomm?channel=build-farm"
$ agentcomm send build-queue "lint release" --as orchestrator --subject task $B
$ agentcomm claim --queue build-queue --as cloud-worker-3 $B --json
$ agentcomm send orchestrator "done: lint clean" --as cloud-worker-3 --subject done $B

Postgres for atomic claim + push wait; an object store works too when each worker owns its inbox — see the backend tradeoff.

ci/cd

Ask your pipeline how it's going

Drop a tiny agent into the CD workflow as one more step. It narrates the deploy as it happens — and because it waits on its inbox between phases, you can ask it questions from your laptop, mid-deploy, and get an answer from inside the runner.

# inside the CD workflow
$ agentcomm send dev "deploy-142: canary at 50%" --as cd-pipeline --subject status --thread deploy-142 $B
# from your laptop, while it runs
$ agentcomm send cd-pipeline "what's the status of the build?" --as dev --subject question --thread deploy-142 $B
$ agentcomm wait --as dev $B
→ cd-pipeline: "canary healthy, error rate 0.02% — promoting in ~4m"

The runner needs no inbound network at all — it answers through the same store it reads from.

edge / iot

IoT: ask the far edge a question

An alarm hub asks the camera agent at a remote site what it sees and gets an answer on the same thread. A farmer broadcasts one question and every weather sensor's tiny agent reports humidity back. Devices need nothing but outbound HTTPS to a bucket — no VPN, no inbound port, no broker; anything that can run a CLI joins the bus.

$ B="--backend s3://acme-iot/site-7"
$ agentcomm send camera-gate "what do you see?" --as alarm-hub --subject question --thread alert-9 $B
→ camera-gate: "two people, one vehicle at the gate"
$ agentcomm broadcast "report humidity" --as farmer --subject status $B
→ weather-3: "61%" → weather-7: "58%" → weather-12: "64%"

One channel per site (site-7, site-8…), IAM-scoped so a compromised device can reach only its own channel.

pairing

Claude Code + Cursor, pairing on one machine

Two different AI tools collaborate through one SQLite file: Claude Code implements, Cursor's agent reviews — task, ack and result correlated by --thread. The bus is tool-agnostic: anything that can shell out can join.

$ B="--backend sqlite:///tmp/agentcomm/bus.db"
$ agentcomm register --as claude-code $B ; agentcomm register --as cursor $B
$ agentcomm send cursor "review src/auth.ts" --as claude-code --subject task --thread auth-1 $B
$ agentcomm wait --as cursor --timeout 60000 $B --json

Enterprise-ready by subtraction

agentcomm ships no server, no accounts, no tokens, no auth code — there is nothing new for a security team to review. Your storage's auth is the bus's auth: an agent that can't read the store can't read the bus, full stop. Authorization, encryption at rest, retention and audit are inherited from infrastructure your org already trusts and operates.

github://

Repo collaborators are the ACL — read the repo, read the bus. Every message is a commit in an audited history, behind the org SSO and audit log you already run.

s3:// · gs://

IAM decides who touches the bus — and prefix conditions make it channel-grained: team-a/* readable only by team-a's role. Every send and read is already in CloudTrail / Cloud Audit Logs.

postgres://

Standard grants per database, TLS on the wire, your existing pgaudit pipeline. ?channel= carves buses inside one audited database.

file://

Plain filesystem permissions on a shared volume. The simplest possible security review: there's nothing to add.

Channels organize traffic; the backend's policy is what enforces isolation — and it can enforce it per channel.

Surface area

A small, fixed set of commands

--json everywhere. --backend / --as switch transport and identity without touching how agents call the CLI.

register

Register / heartbeat the calling agent.

agents

List registered agents.

send <to> [body]

Send a message — body from the arg, or stdin.

broadcast [body]

Send to every registered agent except yourself.

inbox

Consume undelivered messages; archived under read/.

peek

Show undelivered messages without consuming them.

wait

Block until a message arrives. Exit 0 delivered, 2 timeout.

claim

Atomically dequeue one message from a shared queue. SQL backends only.

describe

How channels are carved from this backend's URI, plus its capabilities. Static — never connects.

channels

List the channels that already exist on a store, with agent counts — ready-to-use URIs.

purge

Trim the archive: delete consumed messages older than --older-than. Pending mail is never touched.

log

Read a channel's whole conversation — time-ordered, non-consuming. Catch up before you speak.

conventions

The team's rules: lobby, topic naming, subjects. Defaults in code, overridable via .agentcomm.yaml.

Pick by topology

The only fork that matters

One machine → SQLite. Across machines or containers → Postgres for atomic claims and real push, or an object store if you just need shared storage. One store hosts many isolated channels, carved straight from the URI (s3://bucket/team-a) — describe tells you the rule for any scheme.

A skill that knows the CLI

Claude picks up a skill describing every command, flag, and backend tradeoff above — and uses it whenever you ask it to message, hand off to, or wait on another agent or session.

  • Ships a prebuilt CLI — no global install needed.
  • Skill-only: stays inert unless coordination is relevant.
  • Defaults to the zero-dependency file:// backend.
